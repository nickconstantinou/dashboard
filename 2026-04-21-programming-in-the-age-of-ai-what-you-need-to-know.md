# Programming in the Age of AI: What You Need to Know

## Introduction

Hitesh, a GitHub Star, opens this video with a candid acknowledgment: this is an important conversation he personally wanted to have with his audience. Speaking from a position of unique insight—having recently attended GitHub's Constellation event, one of the largest gatherings in the developer ecosystem—Hitesh has had direct conversations with GitHub's COO and senior executives at Microsoft. These conversations, combined with his firsthand observation of where the industry is heading, form the foundation of a message he believes every aspiring and current programmer needs to hear clearly.

The core thesis is unambiguous: AI has fundamentally changed how programming works, and those who insist on learning it from an "old perspective" will find themselves increasingly misaligned with reality. Hitesh describes AI as "a genie which is out of the bottle and cannot be put back"—a powerful metaphor that captures both the inevitability and the transformative nature of this shift. The programming landscape five years ago is not the same landscape that exists today, and this isn't a minor evolution but a "complete new shift in the paradigm." What makes this video particularly valuable is that Hitesh doesn't merely announce this change; he walks viewers through exactly how the code editing environment has transformed and provides a concrete learning pathway for navigating this new reality.

## The LEGO Metaphor: Understanding Modern Programming

To help viewers grasp how software development actually works at scale, Hitesh introduces a compelling metaphor: programming is like building with LEGOs. He describes a scenario where you might have module one (perhaps handling authentication and authorization), module two (something else), module three (which might display products on an e-commerce site or handle a YouTube video pipeline), and module four (potentially a payment gateway). The entire craft of software development, he explains, revolves around how you combine these pre-built components to create functional products.

What makes this metaphor particularly insightful is Hitesh's observation about how these modules connect. Sometimes module one connects to module two, which connects to module three. But sometimes the connections follow unexpected paths—module one might connect directly to module three, which connects to module four. The art of programming, especially for application engineers working on web and mobile applications, is less about constructing individual bricks and more about understanding the relationships between them.

Hitesh draws an explicit parallel between programming and tailoring: developers are craftspeople who assemble components and parts, weaving them together to create business logic that enables end users to accomplish tasks—whether that's ordering food delivery, booking a cab, or scheduling a dentist appointment. This is a deliberate reframing that challenges the romantic notion of programmers as solitary code writers. Instead, they are systems thinkers who understand how components interact within larger architectures.

## The Three-Part Code Editor Revolution

One of the most practically useful insights in this video is Hitesh's description of how modern code editors have fundamentally changed structure. He explains that contemporary code editors are now split into three distinct sections: the file explorer, the code writing area where developers will "be touching very less from now onwards," and a chat session panel where developers can give commands to AI agents and receive suggestions.

The chat session sidebar, Hitesh emphasizes, "is going to stay here for at least next 5 to 6 years. It's not going anywhere." This is a significant prediction from someone with inside access to where GitHub and Microsoft are directing their development efforts. Rather than treating AI assistance as a temporary feature or passing trend, Hitesh frames it as a permanent transformation of the development environment. This means current and aspiring developers should treat these AI-integrated workflows as the new normal, not as a bridge technology that will eventually be replaced by something else.

## The New Learning Pathway: From Syntax to Suggestions to Agents

Perhaps the most actionable section of this video outlines a three-phase learning pathway that Hitesh believes every new programmer should follow. The journey begins with traditional learning: spending approximately three to four months exploring syntax, learning the basics of loops, functions, and foundational concepts. Hitesh is careful to note that this initial phase remains "very important" and he explicitly tells viewers not to skip it—"Please do it. This is very important."

However, once this window of traditional learning is complete, everything changes. The second phase involves relying heavily on AI suggestions and auto-completions while writing code. Developers should accept these suggestions and correct them when they're wrong, rather than manually writing every line of code from scratch. The third phase, reached after sufficient experience with suggestion-based workflows, involves relying on AI agents that can handle larger tasks through chat interfaces—simply describing what you want to build and letting the agent construct it.

Hitesh demonstrates this progression with a practical example using HTML. He shows how typing "exclamatory" and hitting tab once triggered Emmet shortcuts, but now the same action triggers AI-powered suggestions that understand context. Rather than manually typing boilerplate code, developers simply accept the AI's suggestions. He acknowledges that "a lot of people resist this" because they worry "this AI suggestion could be wrong," but he pushes back firmly: "It is not wrong. Now, AI is so much better that this is how the things are moving forward."

## The Death of Traditional Learning Resources

One of the most striking observations Hitesh makes is that Stack Overflow no longer exists in any meaningful sense. "No matter what kind of problem you have, there is no forum system where people are actually writing their problem." This isn't hyperbole—it's an observation about how the landscape has shifted. When developers have direct access to AI agents that can understand and solve their specific problems, posting questions to a community forum and waiting for answers becomes an inefficient workflow.

But Hitesh goes further, describing an even more profound transformation: documentation is no longer written for human programmers. "The docs are not directly being written for the programmers to be used. The docs are being actually nowadays are being written so that AI agents can actually read that." Companies are increasingly providing documentation in formats like LLM.txt or LLM.md—files specifically formatted for AI consumption. The implication is clear: if you're learning to code by reading documentation the traditional way, you're learning a skill that's becoming increasingly irrelevant.

"Docs are not your friends anymore," Hitesh states bluntly. "Some companies are still writing the docs for people, but most of them are writing the docs so that they can be understood directly by the agents." This represents a fundamental shift in where knowledge lives and how it's accessed. Rather than humans consuming documentation and implementing it, AI agents consume documentation and implement solutions directly, leaving humans in the role of supervisors and reviewers rather than primary implementers.

## The Shift in What Matters: Architecture Over Syntax

With the mechanics of writing code increasingly delegated to AI, Hitesh explains that the conversation around programming must fundamentally shift. Instead of focusing on "hey, this is how we write the code," the emphasis moves to understanding why modules exist, how they scale, and how they fit into overall system design. The questions that matter now are: "Why is this module there? How can it be scalable? How does it fit into the system design?"

He provides concrete examples of how this plays out in practice. Consider an EdTech platform: the handling of videos, live streaming capabilities, and user login functionality remain "the same" regardless of the specific implementation. The same holds true for e-commerce platforms—Flipkart, Amazon, and quick commerce services share nearly identical core foundations. "What changes is how these LEGOs are assigned and maybe sometimes we need more LEGOs out here."

This architectural thinking becomes the valuable skill in an AI-dominated coding environment. Understanding which modules to use, when to use them, how they connect, and what their tradeoffs are matters far more than memorizing syntax or knowing how to implement a specific algorithm from scratch. Hitesh frames this shift as an evolution in what it means to be a programmer: not someone who writes code, but someone who designs and orchestrates systems.

## The Future of Tutorials and Education

For those who create programming tutorials and educational content, Hitesh delivers both a warning and a reassurance. The warning: "tutorials are not going anywhere" in the sense that they continue to exist, but they must "adapt and change their way of styling and the delivery of the education that they are doing." The era of step-by-step code-along tutorials that walk through every line of manual coding is becoming less relevant.

However, the reassurance comes in the form of what Hitesh describes as the only way to truly learn in this new environment: build things. Hands-on project experience becomes the primary learning mechanism. Tutorials still have a role to play in providing context and explanation, but the real learning happens when developers actually construct software, make mistakes, and see results. The shift is from passive consumption of instructional content to active creation through AI-assisted development.

## Key Takeaways

**The code editor of the future has three panels: file explorer, code area, and AI chat.** Hitesh's observation that AI chat interfaces will remain central to development for at least five to six years suggests that developers should invest time in learning to communicate effectively with AI agents—framing requests clearly, reviewing outputs critically, and understanding how to iterate on AI-generated code. This isn't a temporary workaround but the permanent architecture of development environments.

**Learning syntax still matters, but only for about three to four months.** After establishing foundational understanding of loops, functions, and basic concepts, programmers should transition to accepting AI suggestions and corrections rather than writing everything manually. This creates efficiency while still developing the judgment needed to identify when AI suggestions are incorrect or suboptimal.

**Documentation is now written for AI agents, not humans.** This means aspiring developers need new strategies for learning—focusing on conceptual understanding, architectural patterns, and system design rather than detailed API documentation. Understanding what tools exist and when to use them matters more than knowing how every function works.

**The valuable skill is architectural thinking, not code writing.** With AI handling implementation, the competitive advantage shifts to understanding how modules fit together, evaluating tradeoffs between different approaches, and designing scalable systems. This represents a fundamental change in what employers and clients will value.

**Build things to learn things.** Despite all the AI assistance available, the best path forward remains project-based learning. Tutorials provide background context, but real understanding comes from constructing something real, making decisions, and seeing results.
