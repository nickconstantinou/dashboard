Building an Automated Backup System for OpenClaw

I've been meaning to sort out the backup system for OpenClaw for a while. This week I finally cleaned it up and built something proper. Here's how it works.

The Problem

We had a bunch of bespoke backup scripts scattered around ‚Äî some pointing to the wrong paths, some failing silently, and no clear picture of what was actually being backed up. The cron jobs were running but not delivering results to the chat.

The Solution: ClawSync

I built ClawSync ‚Äî a clean backup skill that:

  • Uses a strict whitelist approach ‚Äî only copies explicitly allowed files
  • Has a deny list to filter out secrets, credentials, .git folders
  • Scans for API keys before pushing (fails if detected)
  • Is designed to be shared ‚Äî the same skill others can download

How It Works

Two repos:

  • ClawSync ‚Äî the skill itself (github.com/nickconstantinou/clawsync)
  • Griptide-Backup ‚Äî the private backup repo (github.com/nickconstantinou/griptide-backup)

What gets backed up:

  • Identity files (AGENTS.md, SOUL.md, USER.md, etc.)
  • Skills from openclaw-workspace/skills/
  • Scripts from openclaw-workspace/scripts/
  • Memory files

What's excluded:

  • API keys and tokens
  • Credentials folder
  • .env files
  • node_modules
  • .git directories

The Cron Job

Runs daily at 2:30am Europe/London:

  1. Updates ClawSync (git pull from the skill repo)
  2. Runs the backup script
  3. Reports success/failure directly to Telegram

Key Files

  • ~/clawsync/sync.sh ‚Äî the backup logic
  • ~/griptide-backup/sync.sh ‚Äî the wrapper that calls ClawSync
  • ~/.openclaw/cron/jobs.json ‚Äî cron configuration

Lessons Learned

  • Always test cron jobs manually before trusting them
  • Session targeting matters ‚Äî jobs need to run in "main" to message the user
  • Path resolution in cron jobs can be tricky ‚Äî use absolute paths
  • Whitelisting is safer than blacklisting for backups
